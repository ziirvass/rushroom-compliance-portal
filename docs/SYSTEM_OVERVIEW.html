0. As-Built Status Audited against production 2026-07-07

This section reflects what is actually deployed and live as of 2026-07-07, superseding any "planned/pending" wording later in this document. Frontend is a no-framework static site on GitHub Pages (cache-busted assets, currently ?v=91); backend is the single Supabase portal-api Edge Function (Deno, deployed --no-verify-jwt) plus Postgres + private Storage buckets. The whole document library, action plan, standards register, Level 2 and CELLAR data live in Postgres; the browser never touches the DB directly.

Single AI model. Every server-side AI call — draft generation, deviation scan, clause extraction, interpretations, classification suggestions, directive inference, compliance narrative — uses Anthropic claude-opus-4-8 (constants SCAN_MODEL in index.ts and CLAUDE_MODEL in cellar-service.ts), via api.anthropic.com/v1/messages with adaptive thinking and JSON-schema structured output. No other model IDs are used.

Navigation — six top-level tabs

The portal now has six tabs (in order): Compliance Status, Standards & Regulations, As Operated, Deviation Monitoring, Clauses & DPP, Accounts. Deviation Monitoring and Clauses & DPP are Rushroom-only; Accounts is admin-only. Two views that were once their own tabs are now sub-tabs: the Compliance Map (2×2 board) sits under Compliance Status, and the Directive Graph sits under Deviation Monitoring.

Action plan: "Step" → "Action" (UI wording only)

The action-plan UI now says "Action" throughout (Add action, "Actions by phase", "Delete action #N", the table's Action column). This is a presentation rename only: the database table is still steps, the API verbs are still addStep/updateStep/deleteStep/setStatus, and the text field is transmitted as actionText. Wherever this document says "step", read the UI label "action".

Authentication & Accounts

Individual email + password login (loginUser): PBKDF2-SHA256 hashing (120k iterations), requires a verified email and an approved account; issues an HMAC session token (8-hour TTL) carrying the access tier + admin flag.
Self-registration (role, name, email, phone, WhatsApp, password) → emailed verification link (verify.html); admins review/approve and assign a role in the Accounts tab.
Password reset via signed link (reset.html); admins can copy/resend verify & reset links.
Roles → access tier: admin/internal/reviewer = full ("rushroom") view; supplier/installer = limited ("supplier"). The Accounts tab and admin-only actions (incl. permanent document delete) are gated on the admin flag and marked in red in the UI.
Shared-password login (login) is retained as a bootstrap "super admin" fallback.
Email delivery via Resend (RESEND_API_KEY + MAIL_FROM); admin "Send test email" + status banner. Falls back to copyable links when unset.

A 2×2 classification — lifecycle phase (pre_launch | monitoring) × scope (company | product_services) — now spans documents, action-plan items and clause interpretations. Two Postgres enums plus additive, nullable columns (NULL = unclassified) and an append-only classification_log.

1. System Overview

The Rushroom Compliance Portal is a web-based compliance documentation system that manages the lifecycle of compliance documents, standards, and regulatory requirements for an LED system-furniture product.

Core Concepts

Templates: Reference document types that serve as starting points for operational documents
As Operates: Actual operational compliance documents that reflect current business practice
Standards: Regulatory requirements (e.g., EN 60598-1, CE marking directives)
Versions: Immutable historical records; no documents are ever deleted, only versioned
Provenance: Explicit tracking of which template versions and standard versions were used to create each document version
AI-Assisted Drafting: Claude API generates document updates based on current version + standards + context

2. Data Model

Enums & constraints

PostgreSQL enums: lifecycle_phase (pre_launch, monitoring) and compliance_scope (company, product_services). The classification "quadrant" is the derived (phase × scope) pair — there is no quadrant column.
CHECK constraints: on documents and as_operates_interpretations, the monitoring fields (monitoring_frequency, next_due_at, last_verified_at) must be NULL unless lifecycle_phase = 'monitoring'. Those fields are schema-only for now (no due-date/notification logic yet).
Immutable versioning: document_versions and standard_versions are append-only; document_versions.source_document_version_id and source_standard_version_ids[] carry provenance.
Progressive enrichment: all classification columns are nullable (NULL = unclassified), so a pre-existing Level 1/2 database keeps working unchanged after the additive migration.

7. Key Features & Guarantees

Immutable Versioning

  • Version-controlled: All documents and standards are version-controlled; the normal workflow never deletes — a new version is added instead. Exception: an administrator (super-user) may permanently delete a document via the deleteDocument action, which removes the row (cascading to versions, interpretations and passport links) and the stored files via the Storage API. Finer-grained delete permissions are a later security layer.
  • Full history: Each version row has created_at, uploaded_by, and notes for audit trails.
  • Provenance: Each document_version tracks its source: source_document_version_id and source_standard_version_ids[].

AI-Assisted Drafting

  • Long standard handling: Standards >20 pages are summarized by Claude before drafting to avoid token overflow.
  • Multi-source drafting: AI can read from current document + multiple standard versions + user context simultaneously.
  • Structured output: Claude returns JSON with proposed changes, draft text, version hint, and file name suggestion.
  • Human-in-the-loop: User approves individual changes via checkboxes before publishing.

Standards Register

  • Version tracking: Each standard has multiple versions (2015, 2015+A1, 2015+A1:2022, etc.) with effective dates.
  • Compliance audit: AI can compare all operational documents against the entire standards register in one scan.
  • Deviation findings: Results stored with severity levels (Critical, High, Medium, Low, Info) and actionable recommendations.

Google Docs Integration

  • Seamless editing: After AI draft, user clicks "Edit in Google Docs" to polish the document.
  • Full formatting: Users can add tables, images, comments, track changes, and collaborate.
  • Fetch back: Portal retrieves the edited content and updates the draft before publishing.

Role-Based Access Control

Account roles collapse into two access tiers (roleTier()), and admins carry a super-user flag:

  • "rushroom" tier — assigned roles admin, internal, reviewer. Full read/write on the action plan, documents, standards, Level 2, CELLAR, classification and AI controls. Sees all six tabs (Deviation Monitoring and Clauses & DPP are tier-gated to rushroom).
  • "supplier" tier — assigned roles supplier, installer. Read-only view of supplier-audience actions/documents/standards; may upload files and set status on their own actions. Does not see the Rushroom-only tabs.
  • Admin (super-user) — the admin role (or the shared-password Rushroom login) sets admin: true, which unlocks the Accounts tab and permanent document deletion. Admin-only controls are marked in red.
  • Audience tagging: each action/document/standard has an audience[]; supplier-tier users only ever receive rows whose audience includes supplier.

11. Level 2 Architecture: Structured Clause-Level Interpretations New in Level 2

The system now supports a Level 2 data architecture that decomposes compliance documents into atomic, machine-queryable records. This enables:

  • Direct queries like "what is our interpretation of EN 60598-1 clause 4.11?" without parsing document prose
  • Automated DPP (Digital Product Passport) export in JSON-LD format for EU ESPR compliance
  • Faster, cheaper deviation scanning (using structured data before AI inference)
  • Audit trail with human review tracking at the atomic level

12. EU Directive Relationship Analyser (CELLAR) New module

A dedicated module that maps and visualises how EU directives/regulations relate to each other — at company level (which apply to Rushroom AB as a business) and at product level (which apply to a specific product passport, e.g. the LED wardrobe system). Relationship data is drawn from the EU Publications Office CELLAR repository (public SPARQL + REST, no auth) and, on demand, from Claude AI inference.

Data sources

  • SPARQL endpoint: https://publications.europa.eu/webapi/rdf/sparql — queried for cdm:resource_legal_based_on, _implements, _amends, _repeals and cdm:work_cites_work triples (mapped to relation types requires/implements/amends/supersedes/references, confidence 1.0).
  • REST content API: https://publications.europa.eu/resource/celex/{CELEX} — Akoma Ntoso / XHTML parsed for explicit <ref> cross-references, attributed to the nearest article (akn_ref_element, confidence 1.0).
  • AI inference (on demand only): Claude reads article text and surfaces implicit relations (ai_inferred, confidence 0.0–1.0, always shown in the UI). Never runs automatically — cost control.

New tables (supabase/schema.sql)

  • eu_directives — registry (celex_number, short_name, official_title, directive_type, status, in_force_date, scope_description, eur_lex_url, applies_to_company). Seeded with the 7 directives relevant to the LED wardrobe system (LVD, EMC, RoHS, WEEE, Machinery, Batteries, ESPR).
  • directive_relations — source/target directive links (relation_type, source/target clause refs, source = cellar_sparql | akn_ref_element | ai_inferred, confidence, verified). target_celex is kept even when the target isn't in the registry yet (a gap).
  • product_directive_applicability — directive applicability per product passport (applicable | not_applicable | partial | under_review + rationale).
  • cellar_cache — raw CELLAR results by (celex_number, query_type); re-fetched only if older than 7 days.

All four tables have RLS enabled (deny-all); access is only via the edge function's service role.

CELLAR service (portal-api/cellar-service.ts)

createCellarService(db, anthropicKey) exposes fetchDirectiveMetadata, fetchDirectiveRelations, fetchDirectiveFullText and extractImplicitRelations. All CELLAR access is rate-limited (max 5 concurrent), retried with exponential backoff on 429/503, bounded by a 60s timeout, and 7-day cached. A failure degrades gracefully (null/[]) so the portal keeps working when CELLAR is unreachable.

New API actions (edge function)

  • listDirectives — registry (+ applicability for a passport).
  • addDirective {celexNumber, shortName, appliesToCompany} — fetches CELLAR metadata, upserts.
  • syncDirectiveRelations {directiveId} — SPARQL + AKN <ref> relations upserted; returns {relationsFound, inserted, skipped}.
  • inferDirectiveRelations {directiveId} — on-demand AI inference (never automatic).
  • analyseComplianceGraph {scope, passportId?} — returns {nodes[{id,shortName,status,complianceCoverage}], edges[{source,target,relationType,clauses}], gaps[{celex,reason}]}.
  • setDirectiveApplicability {passportId, directiveId, status, rationale}.
  • getComplianceCoverage {directiveId} — clause-level coverage (total/covered/pct/pending/deviation), tied to standards whose domain matches the directive short-name.
  • generateComplianceNarrative {scope, passportId?, language: sv|en} — Claude writes a human-readable narrative from the graph, for a DoC / passport.

Visualisation (Directive Graph tab, Rushroom only)

A D3.js 7.9.0 force-directed graph (CDN). Nodes = directives, sized by number of applicable clauses, coloured by compliance coverage (green >80%, amber 40–80%, red <40%, grey not assessed). Edges styled by relation type (solid requires/implements, dashed references/amends, dotted AI-inferred) with confidence-scaled width. Click a node → sidebar (title, scope, coverage bar, related directives, EUR-Lex + standards-register links); click an edge → relation type, clauses, description, confidence, source. A Company / Product toggle (product selector in product mode), a gaps table ("Add to portal" per referenced-but-missing directive), and an AI compliance narrative export complete the view.

13. Compliance Status Dimension — Lifecycle × Scope New module

Positions every compliance item (documents + clause interpretations) in a 2×2 matrix: lifecycle phase (pre_launch — one-time pre-market requirements | monitoring — ongoing post-launch obligations) × scope (company — organisation-level | product_services — per-product/service). All columns are nullable (NULL = unclassified): an existing Level 1/2 database keeps working unchanged (progressive enrichment). This phase establishes the dimension and the matrix view only — quadrant-specific behaviour is deferred.

Schema (additive, supabase/schema.sql)

  • Enums lifecycle_phase and compliance_scope.
  • On documents and as_operates_interpretations: nullable lifecycle_phase, scope, classification_ai_generated (bool), classified_by (uuid), classified_at. Interpretations inherit their parent document's classification when their own is unset (overridable per row).
  • Monitoring recurrence — schema only, no logic this phase: monitoring_frequency (interval), next_due_at, last_verified_at, with a CHECK that they are NULL for pre_launch items.
  • classification_log — append-only audit trail of every classification change (entity, old/new phase & scope, changed_by, ai_generated).

Endpoints (edge function)

  • setClassification {entityType, ids[], lifecyclePhase, scope, aiGenerated} — bulk set/update; writes to classification_log. Rushroom/Reviewer only.
  • getComplianceMatrix — pure aggregation (no LLM): per-quadrant totals, compliance_status breakdown, coverage % + colour, and the unclassified count.
  • listClassificationItems — items with own + effective classification; optional lifecycle_phase / scope / unclassified / entityType filters.
  • suggestClassifications — Claude proposes phase + scope for unclassified items (title/category/clause) with a confidence; returns proposals only, never writes. Accepted proposals persist via setClassification with classification_ai_generated = true.

UI — capture at creation + the Compliance Map sub-tab

In the shipped UI, classification is captured at creation time: the new-action and new-document forms include a "Compliance quadrant" picker (lifecycle-phase + scope dropdowns) that flows through as lifecyclePhase/scope on addStep/addDocument. Existing items keep their classification untouched on edit.

The Compliance Map — now a sub-tab under Compliance Status (Rushroom only; the other sub-tab is the action-plan "Status" view) — renders a read-only 2×2 board (renderComplianceMap): a grid of Pre Launch/Monitoring × Company/Product & Services cells, each showing a total, a "N actions · N docs · N clauses" breakdown, a coloured progress bar and % done (green ≥80, amber 40–79, red <40, grey unrated). Clicking a cell lists its items (DOC/ACTION/CLAUSE tags + compliance-status pills); an Unclassified button lists the rest.

Not yet wired: a richer editable classification workbench (renderComplianceMatrix) — quadrant cards, per-item phase/scope selects, multi-select + bulk apply, and an AI "✨ Suggest classifications" flow with per-row Accept/Reject backed by suggestClassifications — is fully implemented in app.js but is not currently mounted into any tab. The live Compliance Map sub-tab uses the read-only board above.

Conclusion

The Rushroom Compliance Portal is a comprehensive, audit-trail-enabled compliance system for the LED system-furniture product. It layers up: Level 1 versioning & provenance → Level 2 structured clause interpretations & DPP readiness → the EU Directive Analyser (CELLAR) → the Compliance Status dimension (Lifecycle × Scope). All four are live in production as of 2026-07-07.

Core (Level 1) capabilities:

  • Immutable document & standard versioning with full audit trail and provenance
  • AI-assisted drafting (claude-opus-4-8) with human review before publish
  • Standards & regulations register with type/jurisdiction, versions and effective dates
  • Client-side Google Docs/Sheets editing round-trip (user's own account)
  • Two-phase AI deviation scan (structured findings first, Claude for the rest)
  • Role-based access (rushroom / supplier tiers + admin super-user), self-registration, email verification

Advanced modules (Level 2 and beyond):

  • Decomposed, clause-level interpretations (AI-assisted or manual), queryable per clause
  • Compliance matrix (docs × clauses) and DPP export in JSON-LD (schema.org + ESPR extensions)
  • EU Directive Relationship Analyser — CELLAR-sourced graph with AI inference, coverage colouring and an EN/SV compliance narrative
  • Compliance Status dimension — Lifecycle × Scope 2×2 classification captured at creation, with the Compliance Map board
  • Progressive enrichment — every classification/interpretation column is nullable, so the system gets smarter without a big migration
Operating the system day-to-day:
  1. Keep the action plan current in Compliance Status; classify new actions/documents by quadrant as you add them.
  2. Extract clauses (extractStandardClauses) and generate interpretations (generateInterpretations) for the main As-Operated documents, then human-review them.
  3. Run the deviation scan periodically; triage findings by severity and the NEW-since-last-scan markers.
  4. Maintain the Directive Graph (sync CELLAR relations, mark applicability per passport) and export the compliance narrative when a DoC/passport is needed.
  5. To deploy changes: push the frontend (bump ?v=N), supabase functions deploy portal-api, and paste schema.sql into the SQL Editor if the schema changed.
